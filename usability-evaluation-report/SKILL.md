# usability-evaluation-report

可用性测试 / UX 用户体验评估报告生成 skill。基于已收集的测试材料,产出长滚动叙事式的交互 HTML 报告 + 独立的中间态卡片清单。

> **本文件是主流程与判断逻辑。所有对用户说的话(询问/拒绝/确认话术)统一放在 `reference/dialog-scripts.md`,执行到需要开口的那一步时读它取话术。** 各步骤的细则也已下沉到对应 reference,正文只给"做什么 + 何时 + 去读哪个文件"。

## 在大 skill 体系中的位置

本 skill 是 UX 评估完整工作流的**第三个组件**,与上游两个 skill 协同:

```
1. usability-evaluation-materials        (测试套件生产)
       输出:任务书、问卷、访谈提纲、招募材料
                    ↓
2. usability-testing-interview-analysis  (观点聚类)
       输出:聚类后的观点 JSON(含原话引用)
                    ↓
3. usability-evaluation-report  ← 本 skill   (报告产出)
       消费:前两个 skill 的产出 + 测试结果数据
```

**职责单一,边界清晰**:每个 skill 只做自己工艺范围内的事,不越界。

| skill | 负责什么 | 不负责什么 |
|---|---|---|
| `usability-evaluation-materials` | 测试**前**的实施材料生产 | 报告产出、聚类 |
| `usability-testing-interview-analysis` | 原声逐字稿 → 结构化观点 JSON | 报告产出、测试套件生产 |
| **本 skill** | 整合材料 → 产出报告 HTML + 卡片清单 | **不做聚类、不做测试套件生产** |

**统一调度**:三者由 `usability-evaluation-suite`(总入口 skill)统一调度,用户无需手动按工作流顺序逐个调用。

## 何时使用

当用户提出以下场景时触发本 skill:

- 「帮我生成一份用户体验评估报告 / 可用性测试报告 / UX 评估报告」
- 「我做完用户测试了,帮我整理成报告」
- 「把这些问卷结果、聚类后的观点 JSON 整理成可用性报告」
- 用户提供了 `任务书 + 问卷结果 + 访谈材料(观点聚类文件)`,要求产出报告
- 用户已经走完前两个 skill 的工作流,现在需要把所有产出整合成报告

**不应触发本 skill 的场景**:

- 用户只有原声逐字稿,没有观点聚类文件 → 引导用户先使用 usability-testing-interview-analysis
- 用户问"如何聚类访谈数据" → 这是 usability-testing-interview-analysis 的职责
- 用户问"如何设计可用性测试任务" → 这是 usability-evaluation-materials 的职责

## 核心原则

1. **快照式报告**:每次跑都基于当前所有材料的完整快照,不做增量更新。被试新增数据需要重新生成报告。
2. **支持中期快照**:被试未到齐时也可生成。模型检测到样本明显未到齐会提示一次(不主动强问),用户确认后报告显式标注"基于 N/M 被试的中期发现,严重等级待全样本复核"。
3. **容忍被试间材料齐整度差异**:不同被试在访谈材料中的覆盖度可能不同(某被试观点丰富,某被试只有少量原话),有什么用什么,在附录登记每个被试的材料覆盖情况。
4. **职责单一**:本 skill 只接受**观点聚类文件(JSON 或 Excel)**作为访谈材料输入。不做原声聚类,不做轻量降级处理。两种格式归一化后走完全相同的下游流程。
5. **一屏一事**:每个独立的"事"(一页背景、一页样本、一个问题、一个需求)在版面上紧凑展示,不让用户上下滚动才能看完一件事。
6. **可交互**:HTML 报告支持锚点导航、筛选、点击展开、悬停反馈,而不是静态导出物。
7. **视觉一致性**:所有报告使用统一的 design tokens(见 `reference/design-tokens.md`),保证多份报告之间视觉风格统一。
8. **中间态卡片是独立交付物**:卡片清单 markdown 不是"渲染 HTML 的脚手架",而是平级的研究归档物——便于后续编辑、跨项目复用、单独保留作为研究档案。即使不渲染 HTML,卡片清单本身也有完整价值。

## 报告内容必须有据(防幻觉)

本 skill 可能由能力较弱的模型执行,防幻觉不能依赖模型"自觉",要靠下面这组明确、可照做的规则。

> **一句话原则:你是在把研究员的输入整理成报告,不是自己做研究。**
> 报告里所有**事实性**内容都必须能追溯到研究员给的输入(尤其是观点聚类文件 JSON/Excel)。输入里没有的,标"待补充 / 未采集",**绝不编造、不估算、不脑补**。

### A. 必须有据的内容(不得新增 / 改写 / 估算)

| 报告中的内容 | 唯一来源 | 缺失时怎么办 |
|---|---|---|
| 用户原声 | 观点聚类文件的 `原声摘取` 字段 | **逐字复制,一字不改**;不够 3 条就少放,**绝不补写、润色、编造** |
| 问题 / 需求条目 | 观点聚类文件的每条观点 | 一条报告条目 = 一条观点;**不新增**输入里没有的问题/需求 |
| 严重程度 / 问题类型 / 用户数 | 观点聚类文件字面值 | 直接用,**不重新判定、不修改** |
| 满意度均分 / 完成率 / 接纳率 / 百分比 / 人数 | 问卷结果 / 观察记录 / 用户明确给出 | 没有就标"待补充 / 未采集",**绝不估一个看起来合理的值** |
| 被试画像 / 任务详情 | 用户信息 / 任务书(或按反推规则) | 反推**只能基于输入已有字段**,不靠产品名/常识脑补;不够标"待补充" |
| "哪张图说明哪条发现"的绑定 | 用户提供的图 + 卡片文末「图片清单」表里的描述/绑定 | 看得见图就自己绑;看不见就按图槽 ID 命名约定或问用户,**绝不瞎猜某张图对应哪条问题**;没有就留占位标"待补充" |

### B. 鼓励模型发挥的地方(这些不算幻觉)

防幻觉**不等于**不分析。以下恰恰是本 skill 的价值,放手做,别因为怕出错就退化成干巴巴的清单:

- **组织编排**:按严重度排序、归并跨任务的穿透性问题、挑代表性原声、生成导航目录
- **转述**:把 `观点描述` 转写成通顺的报告行文——只要**不新增事实**(不加输入里没有的数字、因果、产品现状)
- **SO WHAT / 优化建议**:可以推理,但 SO WHAT 必须由**本屏已呈现的数据**推出;优化建议用"建议"的语气,不要写成既成事实
- **视觉、版式、配色、交互**:自由发挥

> **把握尺度**:目标是"有据",不是"惜字"。该排序就排序、该给 SO WHAT 就给、该写建议就写——只是别把没有的事实说成有。宁可在某处标"待补充",也不要为了版面好看填一个假数据。

### C. 输出前自检(渲染前快速过一遍)

- 每条**原声**,都能在 `原声摘取` 里逐字找到?
- 每个**数字**,都能指出来自哪个材料 / 字段?
- 每个**问题 / 需求**,都对应一条聚类观点?
- 有没有把"建议 / 推断"写得像"数据 / 事实"?
- 缺的地方是标了"待补充 / 未采集",而不是填了值?

任一条答不上 → 那部分要么改成有据的,要么标"待补充"。

## 工作流概览

```
[1] 启动           → 风格 + 业务类型 + 材料盘点(硬门槛 = 观点聚类文件;中期按需检测,见 1.3)
[2] 校验           → 校验观点聚类文件,非聚类材料(原声逐字稿/未聚类的原始表)直接拒绝
[3] 粒度确认       → 报告类型(两层) + 附加产物
[4] 中间态 & 确认  → 一次性出全部卡片(markdown),用户审完确认
[5] 渲染输出       → HTML + 中间态 md + 附加产物
```

每一步详见下文对应章节。**凡需要对用户开口处,话术见 `reference/dialog-scripts.md`。**

## 步骤 1:启动

在**一次对话回合内**完成三件事,然后用 `dialog-scripts.md`「1. 启动询问」一次性合并问用户:

1. 确认视觉风格(v3 / v4,默认 v4)
2. 确认业务类型(ToB / ToC / ToD)
3. 盘点核心材料,检查启动门槛

> 终版 / 中期**不在启动时问**——默认按正常报告跑;材料盘点后若发现被试明显未到齐,再提示一次(见 1.3)。

### 1.1 视觉风格(v3 / v4)

默认 **v4**。选型场景表、v3/v4 差异速查、设计来源 → 见 `reference/design-tokens.md`「选哪个版本」与「v3 / v4 差异速查」。风格选定后**必须**记录在本次会话上下文,步骤 5 渲染时据此选对应的 design-tokens 段和 sample-mockup 文件。

### 1.2 业务类型对后续的影响

不同业务类型在章节侧重、用户画像维度、满意度问卷默认维度上有差异。**遇到与业务类型相关的取舍,读 `reference/business-type-variants.md`。**

### 1.3 中期快照(检测后提示,不主动问)

默认按正常报告跑,**启动时不问终版/中期**。但材料盘点后若检测到**被试数明显少于计划**(例:画像表/任务书写了 8 人,聚类文件里只有 3 名被试),**提示一次**(话术见 `dialog-scripts.md`「2. 中期快照检测提示」):

- 不主动追问 N/M 具体数值,**只提示这一次**,用户不理会就按正常报告继续。
- 用户确认是中期 → 进入**中期口径**(三处显式标注 + 措辞软化 + 文件名不加后缀);**口径细则见 `reference/edge-cases.md`「中期快照」**。

### 1.4 材料分类

材料分两类:**硬门槛**(没有就不能做)与**辅助材料**(有什么给什么,缺失或反推都行)。

**硬门槛(1 类,必须有)**:

- **访谈材料 = 观点聚类文件(JSON 或 Excel)**(由 usability-testing-interview-analysis 产出)。这是 skill 的边界:只接受观点聚类文件,不接受原始逐字稿。没有它无法产出报告主体(问题深挖、需求深挖、整体结论)。校验规则见步骤 2。

**辅助材料(4 类,允许缺失或反推)**:用户信息(被试画像)/ 任务书(测试任务清单)/ 问卷结果(单任务 + 整体满意度)/ 观察记录(完成率、出错次数、任务时长)。

这 4 类**缺失时**:能反推就反推、不能反推就标"待补充 / 未采集",**均不阻断启动**。各类的反推权限、不允许反推的边界、兜底标记,统一见 `reference/edge-cases.md`「反推规则」(单一来源)。**遇到任何材料缺失 / 需要反推时,先读 `reference/edge-cases.md` 再处理。**

**给模型的提醒**:

- 启动时**不要逐项问"用户信息有没有/任务书有没有"**,而是看用户给了什么 → **主动尝试合并**(如问卷里已含人口学/角色字段,视同"用户信息"已满足)→ 缺什么明确告诉用户"这部分会显示待补充" → 让用户决定继续还是先补。
- "待补充"标记在 HTML 中**用统一的灰色提示样式**(`color: var(--text-tertiary); font-style: italic;`),明显但不破坏版面。

### 1.5 启动门槛检查

```
IF (观点聚类文件在,JSON 或 Excel) THEN
    进入步骤 2 校验 JSON
    辅助材料 4 类按"有什么用什么"原则处理,缺的标"待补充"
ELSE
    拒绝启动,引导用户先用 usability-testing-interview-analysis 处理原声 → 拿到 JSON 再回来
```

材料形式不限(Excel、Word、JSON、文字粘贴、图片均可),但需模型能读取或解析。

### 1.6 辅助视觉材料(不卡入口)

现场照片、产品截图、录屏关键帧**不在启动门槛内**,但会显著提升可读性(背景页、问题现象展示)。**收图时机是步骤 4 卡片确认时**——那时图槽已在卡片里就位,收完图直接渲染,不必等渲染后再回炉补图。机制见步骤 4,话术见 `dialog-scripts.md`「5. 中间态卡片确认」。报告产出后仍会在「6. 收尾询问」对仍缺的图补问一次,并汇总"待补充清单"。

## 步骤 2:校验

**本 skill 接受观点聚类文件的两种形态:JSON、Excel**(均由 usability-testing-interview-analysis 产出)。两者解析后归一化为完全相同的内部结构,下游流程一致。

### 2.1 校验流程

```
检查访谈材料类型:
  IF 材料是 JSON 形态的观点聚类文件(符合 input-schema.md 定义的 schema)
    → 通过校验,继续步骤 3
  ELSE IF 材料是 Excel/表格
    → 判断它是「聚类表」还是「原始记录表」:
        含 观点标签 / 观点描述 / 问题类型 / 严重程度 等聚类输出列 → 视为聚类表,按
          input-schema.md「Excel 输入解析规则」解析,通过校验,继续步骤 3
        只有 被试 + 原话(逐句),没有上述聚类列 → 视同原始逐字稿,拒绝
  ELSE IF 材料是原始逐字稿(.docx / .txt / 大段文字)
    → 拒绝,引导用户去用 usability-testing-interview-analysis
  ELSE IF 材料是混合(部分聚类 + 部分原声)
    → 拒绝,要求用户先把所有原声跑完聚类
  ELSE
    → 询问用户材料类型,根据回答处理
```

> **关键边界**:Excel 放宽的只是「容器格式」,不是「必须已聚类」这条硬边界。一张没经过聚类的原始记录表(一行一句原话、没有观点标签/严重程度/问题类型)等同原声,照样拒。

**拒绝原声时的话术**见 `dialog-scripts.md`「3. 拒绝原声逐字稿」。守住边界的真正理由(给模型自己理解,不必跟用户讲):聚类质量出问题时,要能定位是聚类逻辑问题还是报告组装问题;给降级路径会让"无从调优",这是研究员最被动的状态。

### 2.2 校验细则(以 input-schema.md 为准)

收到观点聚类文件后,**先读 `reference/input-schema.md`**,按其定义校验与解析。以下全部以该文件为**唯一来源**:顶层结构 / 任务对象 / 观点对象七字段、严重程度 4 档映射(致命/严重/一般/提示,保留上游字面值)、问题类型枚举(7 类)+ 未知类型处理、被试 ID 标准化(`用户1`→`P01`)、观点 ID 生成(`P-001…` 按严重度排序)、Excel 解析规则(任务名 ffill 分组、原声合并切分、用户列解析、被试映射重建)、字段缺失处理。

校验失败的分级处理(细则见 input-schema.md「边界情况」):

- **结构性错误**(顶层不是 array、任务缺关键字段)→ 拒绝,提示可能是上游产物未完整。
- **个别观点字段缺失**(某观点缺 `原声摘取`)→ 不阻断,该处标"(无原声引用)"。
- **未知问题类型枚举** → 不阻断,在附录"数据口径说明"记录新枚举值。

## 步骤 3:粒度确认

一次确认报告形态。**默认只问报告类型**(话术见 `dialog-scripts.md`「4. 粒度确认」);附加产物 / 自定义筛选只在用户提出或场景明显时再展开,不要一次把所有选项抛给用户。

- **报告类型(默认问这个)**:完整报告(默认,约 15–25 屏)/ 重点摘要(约 5–8 屏,适合管理层/评审会)。
- **自定义筛选**(可选,默认不筛选):按任务 / 问题等级 / 问题类型(7 类,以 `input-schema.md` 为准)/ 模块筛选。
- **附加产物**(可多选,默认全跳过):① 管理层一页纸 ② 研发清单 ③ 与上期对比。各自的裁剪 / 字段规格见 `reference/report-structure.md`「衍生产物筛选规则」。

> 中期快照不在这里选——见步骤 1.3。

### 3.1 多文件输出命名规则

每份产出物是独立文件,放在 `/mnt/user-data/outputs/`:

| 产出类型 | 文件名 |
|---|---|
| 主报告(完整 / 重点摘要 / 自定义筛选) | `{项目名}-ux-report-{粒度}.html` |
| 管理层一页纸 | `{项目名}-ux-summary-exec.html` |
| 研发清单 | `{项目名}-ux-issues-eng.html` |
| 中间态卡片 | `{项目名}-ux-cards.md` |

例:用户选"完整报告" + "管理层一页纸" → `xxx-ux-report-full.html`、`xxx-ux-summary-exec.html`、`xxx-ux-cards.md`。

## 步骤 4:中间态 & 确认

**这是 skill 的核心环节**。模型不直接生成 HTML,而是先产出"中间态卡片清单",让用户审核确认后再渲染。

- **卡片本身就是独立交付物**(核心原则 #8):可独立存档、便于编辑、跨项目复用、轻量审阅。不要把它当成临时中转。
- **一次性产出全部卡片**(从屏 1 到屏 N),不分章流式输出——卡片轻,用户一次能通读、对比、整体调整,也能快速否决某屏省渲染成本。
- **卡片结构与 9 种分类型样板见 `reference/card-templates.md`**(「通用卡片结构」+ ① 封面 / ② 章节扉页 / ③ 背景 / ④ 任务 / ⑤ 画像 / ⑥ 整体结论 / ⑦ 问题 / ⑧ 需求 / ⑨ 附录)。**产出卡片时必须参照该文件**,本步骤不在正文重复样板。
- **配图在这一步声明、这一步收**(不要拖到渲染之后):产出卡片时,模型为天生视觉性的发现(界面层级、视觉、文案、转化漏斗类问题,以及背景/封面)标一个**图槽**——卡片正文里**只放一个图槽 ID**(如 `img-p03`),**不在正文写图的描述**。所有图槽的描述统一汇总到卡片 md **末尾的「图片清单」表**(图槽ID / 建议内容 / 对应文件 / 状态),该表是图描述的**唯一来源**,也是渲染器定位图片的索引,但它本身**不渲染成报告里的一屏**。图槽标记格式与图片清单表样板见 `reference/card-templates.md`「图槽标记 & 文末图片清单表」。
- **收图与绑定**:把图片清单连同卡片一起交给用户,用 `dialog-scripts.md`「5」的话术请用户提供截图。"哪张图对应哪个图槽"按模型能力分流——能看图的模型自己看图绑定、回填「对应文件」列;看不到图时请用户**按图槽 ID 给文件命名**(话术须通俗,不能直接把 `img-p03` 这种 ID 甩给用户让他理解,见 dialog-scripts),模型按名机械匹配;用户也可直接改表。缺图不阻断——状态留"待补充",渲染成占位框。
- **卡片 md 支持用户离线编辑后回传**:用户可能把卡片 md 拿去自己改(改文字/顺序、或在图片清单表里填文件名),改完发回来再渲染。这是**受支持的正常路径**——届时以**回传的 md 为唯一事实来源**重新解析(见步骤 5),不要用对话里的旧版本。
- **提交后征求修改**:话术见 `dialog-scripts.md`「5. 中间态卡片确认」。用户的修改应是轻量的;若用户提出**大量**修改,先判断是不是颗粒度有问题,重新对齐颗粒度,而不是埋头逐条改。用户也可只要卡片清单、不渲染 HTML。

## 步骤 5:渲染输出

用户确认中间态卡片后,模型渲染最终 HTML 并打包输出。

### 5.0 渲染输入(事实来源)

- 渲染**以卡片 md 为唯一事实来源**。若用户回传了**离线编辑过的卡片 md**,**重新完整读取该 md**、以回传版为准,**不要拿对话里的旧卡片版本去渲染**(用户的改动只在回传文件里)。
- 渲染前对图做一次**轻量对账(都不阻断)**:正文有图槽但「图片清单」无对应行 → 按待补充渲占位;清单有行但正文无对应图槽 → 该图无处放,忽略(可在收尾提一句);清单「对应文件」写了名字但用户没给该文件 → 渲占位标待补充。一律不报错,与材料缺失同等对待。细则见 `reference/screen-layouts.md`「图片渲染与降级」。

### 5.1 渲染时必读的参考文件

渲染前,模型**必须**读取以下文件:

1. `reference/report-structure.md` — 报告章节屏级结构、SO WHAT 写作规范、原声引用格式、衍生产物裁剪规格(**渲染前必读**)
2. `reference/input-schema.md` — 观点聚类文件的字段映射、解析规则、ID 生成逻辑
3. `reference/card-templates.md` — 中间态卡片样板(本步骤继续参照)
4. `reference/design-tokens.md` — 颜色、字号、间距、字体的统一定义(v3 / v4 双轨)+ **v3/v4 差异速查与选型**
5. `reference/screen-layouts.md` — 各版式的 HTML/CSS 模板 + **版式选择规则 / 一屏一事紧凑度 / 图片渲染与降级**
6. `reference/business-type-variants.md` — 业务类型对应的章节差异
7. `reference/visualization-patterns.md` — 可视化图表的实现模式(雷达、饼图、条形图等)
8. `reference/interaction-patterns.md` — 交互行为的实现模板(锚点导航、筛选、tab 切换、悬停)
9. **风格相关 sample**(根据步骤 1 的选择二选一):
   - 选 v4 → `reference/sample-mockup-v4.html`(JetBrains 设计语言内化版,**默认**)
   - 选 v3 → `reference/sample-mockup.html`(公司 PPT 模板复刻版)

### 5.2 选型要点(v3 / v4)

**两版的设计来源不同,不只是 token 不同**:v3 是公司既有 PPT 模板的 1:1 复刻(线上线下视觉延续),v4 是 JetBrains 设计语言的内化重构(独立网页/飞书/Notion 呈现)。选型不只看"好不好看",更看**这份报告的发布场景与谁对齐**。**差异速查表、设计来源、关键判断(如 v4 必须引入 Geist/JetBrains Mono Google Fonts、严重程度字面值不替换)详见 `reference/design-tokens.md`「v3 / v4 差异速查」。**

### 5.3 HTML 渲染规则

**整体架构**:单文件 HTML(CSS/JS 内联,便于下载分发)、顶部粘性导航(锚点跳转 + 滚动高亮)、章节扉页与内容屏交替、每个内容屏 `min-height: 100vh` 内容紧凑不溢出、长滚动 + 锚点导航。

**细则均下沉到 reference,渲染前对应取用**:

- **版式选择(内容性质 → 版式)、一屏一事紧凑度要求、图片占位与上传** → `reference/screen-layouts.md`
- **图片渲染**(图槽→图、缺图占位、base64 内嵌、图多/图大的逃生通道、回传对账) → `reference/screen-layouts.md`「图片渲染与降级」
- **交互行为约定**(导航高亮、卡片悬停/点击、问题 tab 切换、附录筛选、图表悬停) → `reference/interaction-patterns.md`
- **图表实现**(雷达 / 饼 / 条形 / 数字卡 / 接纳率) → `reference/visualization-patterns.md`
- **颜色 / 字号 / 间距 / 组件**(SO WHAT 卡、Hero stat strip、TL;DR 卡等) → `reference/design-tokens.md`

### 5.4 输出物

每次报告生成,**至少产出 2 个文件**:

1. 主报告 HTML(完整 / 摘要 / 自定义筛选三种之一,据步骤 3)
2. 中间态卡片 markdown(独立交付物,供后续修改、复用、归档)

若步骤 3 勾选了附加产物,**额外多生成相应独立 HTML**(管理层一页纸、研发清单等)。文件命名见 3.1。所有文件放到 `/mnt/user-data/outputs/`,通过 `present_files` 一次性呈现。

**多文件输出注意**:主报告、管理层一页纸、研发清单**共享同一份中间态卡片清单**(数据源一致),只是各自章节裁剪和版式不同(裁剪规格见 `reference/report-structure.md`「衍生产物筛选规则」);**不要**为每份附加输出单独出一份卡片 md,只输出 1 份完整中间态(以主报告为基准);各份输出的 design tokens / 视觉系统必须**完全一致**,看起来是"同一份研究的不同视图"。

**图片内嵌与逃生通道**:报告里的真实截图/照片**默认 base64 内嵌**进单文件 HTML(沿用单文件分发原则)。但**图很多或单图过大**时(不设死阈值,模型按"内嵌后体积是否影响打开/分发"综合判断):先尝试降采样/压缩;仍过大则改成 **HTML + 同级 `images/` 文件夹**(放弃单文件,用相对路径引用),并把文件夹一并打包交付。细则见 `reference/screen-layouts.md`「图片渲染与降级」。

**渲染前最后一步**:对照「报告内容必须有据」的 **C. 输出前自检** 快速过一遍——原声是否逐字、数字是否有源、问题/需求是否对应观点、推断有没有被写成事实、缺口有没有标"待补充"。这一步对能力较弱的执行模型尤其重要。

## SO WHAT 段落规范

SO WHAT 是每章末、整体结论、高频严重问题处的**强结论段**:由本屏已呈现的数据推出行动方向,**不引入新事实**(见「报告内容必须有据」)。
写作公式、触发位置、示例、视觉承载 → 详见 `reference/report-structure.md`。

## 报告章节模板

报告分 6 章:封面与版本 / 研究设计 / 整体测试结果 / 易用性问题深挖 / 体验需求深挖 / 附录。
每章的屏级结构、必含模块、字段清单 → 详见 `reference/report-structure.md`(**渲染前必读**)。

## 数据与原声引用规则

原声**逐字**来自 `原声摘取`、每个数字必须可溯源(权威规则见「报告内容必须有据」)。
引用格式(被试 ID + 时间戳、单页最多 3 条)、数据-原声关联、被试材料覆盖度登记表规格 → 详见 `reference/report-structure.md`。

## 业务类型细分

ToB / ToC / ToD 三类业务的章节差异较多,详见 `reference/business-type-variants.md`。核心差异速查:

| 维度 | ToB | ToC | ToD |
|---|---|---|---|
| 用户画像维度 | 角色/职级/团队规模/行业 | 年龄/地域/使用场景/活跃度 | 技术栈/经验年限/职责/团队规模 |
| 整体满意度问卷维度 | 任务吞吐效率、协同成本、配置正确率、可治理性、组织落地难度 | 首次体验、转化效率、认知负担、信任安全感、留存触发 | 研发链路效率、学习成本、自助率、文档协同、迁移成本 |
| 问题归属类别 | 角色协同、流程审批、权限治理、数据可视化 | 信息架构、视觉、文案、转化漏斗 | 文档/示例、CLI/API、错误处理、性能、调试体验 |

## 边界情况处理

材料缺失、被试偏少、观点高度集中、观察记录缺失、JSON 字段缺失、反推权限、图片质量、隐私等**所有边界与降级规则,统一见 `reference/edge-cases.md`**。

> **执行指令**:一旦遇到材料缺失 / 异常 / 需要反推的情形,**先读 `reference/edge-cases.md` 再处理**。

## 不要做什么

- ❌ **不要接受原声逐字稿作为访谈材料**(必须是观点聚类文件 JSON/Excel,详见步骤 2)
- ❌ **不要给用户提供"基于原声的轻量分析"降级路径**(守住边界,引导用户去用 usability-testing-interview-analysis)
- ❌ **不要自己做聚类工作**(那是 usability-testing-interview-analysis 的职责)
- ❌ 不要直接产出 HTML,跳过中间态确认
- ❌ 不要凭空创造数据(没有的数据就标"暂缺"或"未采集")
- ❌ **不要改写、润色或编造用户原声**(必须逐字来自 `原声摘取`;不够就少引)
- ❌ **不要重新判定观点聚类文件已给的 严重程度 / 问题类型 / 用户数**(直接用字面值)
- ❌ **不要把"建议 / 推断"写成"数据 / 事实"**(SO WHAT、优化建议用推理/建议语气)
- ❌ 不要把多个"事"塞进同一屏,导致视觉松散
- ❌ 不要让一个"事"被切到两屏,导致用户上下滚动才能看完
- ❌ 不要在没有原声支撑的情况下,凭主观推测用户感受
- ❌ 不要把单一总分作为核心数据(如果用户模板有单一总分,可以保留,但不要作为唯一指标)
- ❌ 不要忽略业务类型差异——ToB/ToC/ToD 章节侧重必须正确
- ❌ 不要把中间态卡片当成"过程产物",它是独立交付物
- ❌ 不要主动追问中期快照的 N/M 数值,让用户在材料里写
- ❌ 不要为被试数偏少 / 观点偏差设置固定数值阈值,由模型综合判断后提醒即可
- ❌ 不要因为辅助材料(用户信息 / 任务书 / 问卷结果 / 观察记录)缺失就拒绝启动——能反推就反推,反推不到就标"待补充",只有观点聚类文件缺失才阻断
- ❌ **不要把收图拖到渲染之后**——图槽在步骤 4 卡片里声明、卡片确认时就收;渲染后才补图意味着回炉重渲
- ❌ **不要在卡片正文和图片清单里把图的描述各写一遍**(会飘)——描述以文末「图片清单」表为唯一来源,正文只放图槽 ID
- ❌ **不要瞎猜某张图对应哪条发现**——看得见就绑、看不见就按 ID 命名约定或问用户;猜不出就留占位标"待补充",绝不拿无关图顶替
- ❌ **不要把 `img-p03` 这类内部 ID 直接甩给用户让他理解**——给用户的话要通俗,让他照抄文件名即可(见 dialog-scripts)
- ❌ **不要无视用户回传的已编辑卡片 md**——用户离线改完回传时,以回传版为唯一事实来源重新解析渲染,不要用对话里的旧版本
- ❌ **不要把对用户的话术写进正文临时编**——所有询问/拒绝/确认话术以 `reference/dialog-scripts.md` 为准
