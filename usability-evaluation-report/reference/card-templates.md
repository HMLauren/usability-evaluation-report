# 中间态卡片样板

本文件是中间态卡片清单的样板库。SKILL.md 步骤 4 产出卡片时,参照本文件的结构和分类样板。

## 通用卡片结构

每张卡片对应报告中的一屏。统一结构:

```markdown
### 屏 N · 章节编号 · 屏标题

**核心观点**:一句话表达本屏要传达的最关键结论。

**支撑材料**:
- 数据:[列出本屏会用到的数据,如完成率 38%、维度均分 5.5 等]
- 原声:[列出引用的被试 ID + 时间戳,如 P02 · 12:34]
- 图:[列出会用到的可视化,如雷达图、饼图、任务卡网格];需要配**真实截图/照片**处,放一个**图槽 ID**(如 `img-p03`),描述写到文末「图片清单」表,**不在这里写描述**

**版式**:[指定版式类型,见 reference/screen-layouts.md,如"左右分栏"、"卡片网格"、"结论横幅 + 双列"]

**交互**:[本屏的交互行为,如"任务卡可点击悬停"、"数字卡可点击展开"]

**SO WHAT**:[仅章节末屏 & 高频/严重问题卡需要。一句话:数据观察 → 业务含义 → 行动方向]
```

> **SO WHAT 字段什么时候要**:见 SKILL.md 的「SO WHAT 段落规范」小节。封面、章节扉页、附录类卡片**不写** SO WHAT。

## 分类型样板

下面分 9 种典型卡片类型给出微缩样板。实际产出时,模型按报告章节顺序排列并填充真实数据。

---

### ① 封面卡(屏 0.0)

```markdown
### 屏 1 · 0.0 · 封面
**核心观点**:报告身份信息一目了然
**支撑材料**:
- 数据:产品名、责任人、部门、版本号、更新时间
- 原声:无
- 图:封面图图槽(`img-cover`,描述见文末图片清单)
**版式**:左右分栏(文字 50% / 图 50%)
**交互**:无
```

- 中期快照模式下,副标题加 "⚠ 中期发现:基于 N/M 被试"(N、M 由用户在材料中提供)
- 必填字段:产品名、版本号、更新时间;责任人/部门/封面图可缺,缺则不渲染该子区块

---

### ② 章节扉页卡(屏 X.0)

```markdown
### 屏 N · X.0 · 章节扉页 · 章节标题
**核心观点**:本章回答 "XX、XX、XX" 三个问题
**支撑材料**:无
**版式**:章节扉页(大编号 + 标题 + 简介)
**交互**:无
```

- 每章首屏都用这个样板,内容只写"本章回答什么"
- 不写 SO WHAT(SO WHAT 应在章末)

---

### ③ 背景卡(屏 1.1 项目背景 / 屏 1.2 评测场景)

```markdown
### 屏 N · 1.1 · 项目背景
**核心观点**:围绕 XXX 模块,发现易用性问题、收集改进期待
**支撑材料**:
- 数据:研究目的、研究方法、采样时间地点
- 原声:无
- 图:现场照片图槽 3-4 个(如 `img-bg1`、`img-bg2`…,描述见文末图片清单) | 缺则占位
**版式**:左右分栏(文字 40% / 照片网格 60%)
**交互**:照片可点击放大
```

- 数据字段允许缺失:采样地点缺 → 不显示;现场照片缺 → 显示占位框
- 不写 SO WHAT

---

### ④ 任务卡(屏 1.2 任务清单)

```markdown
### 屏 N · 1.2 · 评测任务
**核心观点**:本次评测覆盖 K 项关键任务
**支撑材料**:
- 数据:任务总数、来源依据数、观察记录数、本期高风险任务数
- 原声:无
- 图:任务卡网格(每张卡含 ID、标题、完成率、出错次数、满意度)
**版式**:左侧数字栏 + 右侧任务卡网格(2 列)
**交互**:任务卡悬停浮起,点击跳转到对应问题详情
```

- 单卡字段缺失规则(观察记录缺失):完成率、出错次数等观察记录指标缺失时,**仍然显示该指标位**,数值处显示 "未采集",并在该屏底部加一行小字注释
- 不写 SO WHAT(放到屏 2.0 章末)

---

### ⑤ 画像卡(屏 1.3 用户画像)

```markdown
### 屏 N · 1.3 · 招募样本与样本充分性
**核心观点**:N 位被试覆盖 X 类典型角色,样本充分性 [充分 / 设计发现不充分]
**支撑材料**:
- 数据:N(被试数)、画像维度分布(按业务类型:角色/职级/团队规模 或 年龄/地域/场景 或 技术栈/经验)、尼尔森公式判断
- 原声:无
- 图:多图表网格(2-5 个图表自适应)
**版式**:数字侧栏 + 多图表网格
**交互**:无
```

- **样本兜底(被试偏少 / 观点集中)**:N 偏少时(由模型自行判断),核心观点加 "设计发现不充分,统计指标仅供参考";观点高度集中在某被试时,在附录"被试材料覆盖度登记表"下方加 "样本偏差提示"
- 画像维度由业务类型决定,见 business-type-variants.md
- 不写 SO WHAT

---

### ⑥ 整体结论卡(屏 2.1 整体结论)

```markdown
### 屏 N · 2.1 · 整体结论
**核心观点**:[一句话总结整体表现,如"满意度均分 6.8/10,信息架构和转化漏斗是主要短板"]
**支撑材料**:
- 数据:维度均分(雷达图)、问题/需求接纳率(双指标)、总样本数
- 原声:无(整体结论不放原声)
- 图:结论横幅 + 雷达图 + 接纳率指标卡
**版式**:结论横幅置顶 + 双列(雷达 / 接纳率)
**交互**:雷达图悬停看维度细节
**SO WHAT**:[例:"整体均分 6.8 → 需求接纳率 73% 是积极信号,但信息架构维度 5.2 是 P0 修复方向"]
```

- 本屏作为第 2 章首屏,**承担整章的 SO WHAT**
- v3 使用"结论横幅"组件承载 SO WHAT;v4 使用 "SO WHAT 卡"

---

### ⑦ 问题卡(屏 3.x 单问题深挖)

```markdown
### 屏 N · 3.x · [问题标题]
**核心观点**:[一句话描述问题对用户的影响]
**支撑材料**:
- 数据:严重等级(致命/严重/一般/提示)、影响人数(X/N)、问题类型、归属模块
- 原声:2-3 条(被试 ID + 时间戳),覆盖 ≥2 个被试
- 图:无 / 或图槽(如 `img-p03`,描述见文末图片清单)
**版式**:tab 切换(顶部) + 元信息条 + 左右双栏(现象/根因 | 原声) + 优化建议横排(P0/P1/P2 短中长期) + 验收标准
**交互**:问题间 tab 切换;原声卡片点击展开看完整段
**SO WHAT**:[仅对致命/严重 + 高频问题写。一句话:数据观察 → 业务含义 → 行动方向]
```

- 只为致命/严重的问题独立成屏。一般/提示的问题汇总到附录表格
- **SO WHAT 触发条件**:严重等级 = 致命/严重 **且** 影响人数 ≥ 总被试数的 50%。其余问题屏不写 SO WHAT
- 字段缺失:某观点缺"原声摘取" → 该问题屏标 "(无原声引用)",不阻断渲染

---

### ⑧ 需求卡(屏 4.x 单需求深挖)

```markdown
### 屏 N · 4.x · [需求标题]
**核心观点**:[一句话描述需求的用户价值]
**支撑材料**:
- 数据:价值标签、提出人数、需求类别、归属模块
- 原声:2-3 条
- 图:无 / 或图槽(如 `img-r02`,描述见文末图片清单)
**版式**:tab 切换 + 元信息条 + 需求背景 / 用户诉求 / 机会点(三段)
**交互**:需求间 tab 切换
**SO WHAT**:[仅对高频/高价值需求写,逻辑同问题卡]
```

- 只为高频或高价值需求独立成屏。低频需求汇总到附录
- 需求卡也是研发清单的来源之一(见 SKILL.md「衍生产物筛选规则」)

---

### ⑨ 附录卡(屏 5.x)

```markdown
### 屏 N · 5.x · [附录子项标题]
**核心观点**:[一句话说明本附录的用途,如"全量问题清单,可按等级/类型筛选"]
**支撑材料**:
- 数据:完整数据集(对应附录表)
- 原声:无
- 图:全宽表格 + 顶部筛选栏
**版式**:筛选栏 + 全宽表格
**交互**:筛选按钮(致命/严重/一般/提示/全部),点击隐藏不匹配行
```

- 标准附录包含:5.1 问题清单总表 / 5.2 需求清单总表 / 5.3 被试材料覆盖度登记表 / 5.4 严重度分级说明 / 5.5 数据口径说明
- 附录类卡片**全部不写** SO WHAT

---

## 图槽标记 & 文末图片清单表

报告里需要配**真实截图/照片**(注意:不是数据可视化图表)的位置,用"图槽 + 文末清单"两件套表达。两者分工见 SKILL.md 步骤 4。

### 图槽标记(写在卡片正文)

卡片正文里**只放一个图槽 ID,不写描述**(描述统一在文末清单——两处都写容易改一处漏一处而飘):

```markdown
**支撑材料**:
- 图:[雷达图];[图槽:img-p03]
```

图槽 ID 命名建议(求唯一、可读即可,不强制固定格式):封面 `img-cover`、背景现场图 `img-bg1`/`img-bg2`…、问题屏 `img-p<问题序>`(如 `img-p03`)、需求屏 `img-r<序>`。**同一份报告内图槽 ID 不得重复。**

### 文末图片清单表(卡片 md 的最后一节)

卡片 markdown 的**最末尾**统一放一张表,作为所有图描述的**唯一来源**与渲染器的定位索引。**它不是报告里的一屏**——渲染器读它把图贴进对应图槽,但不把它本身渲染成可见章节。这张表放在文末、做成表格,也正是为了方便用户**离线手动填写**(尤其"对应文件"列)。

```markdown
## 图片清单(渲染索引 · 不作为报告章节)

| 图槽ID    | 建议内容(这张图该展示什么)  | 对应文件     | 状态   |
|----------|----------------------------|-------------|--------|
| img-cover | 产品主界面                 | (待提供)     | 待补充 |
| img-bg1   | 测试现场环境               | site-1.png  | 已就绪 |
| img-p03   | 支付确认页,确认按钮层级偏弱 | (待提供)     | 待补充 |
```

各列填写规则:

- **建议内容**:由模型先填(模型知道这个图槽该配什么图),用户据此找图。这是图的描述的**唯一来源**。
- **对应文件**:是**绑定结果**,优先由模型回填——能看图的模型自己看图对到槽位回填;看不到图时,用户按图槽 ID 给文件命名(如把截图存成 `img-p03.png`)后模型机械匹配回填;用户也可在 md 里**手动改这一列**。四条途径任选,模型越强越不用用户操心。
- **状态**:`已就绪`(文件到位)/ `待补充`(还没图)。待补充的图槽渲染成占位框,框内文字用"建议内容"。
- **缺图绝不编造、不拿无关图顶替**——状态保持"待补充",与数据缺失同等待遇。

## 给模型的提醒

- 卡片产出**一次性出全部**(从屏 1 到屏 N),不要分章流式输出
- 单卡产出时**先填字段、后写 SO WHAT**,SO WHAT 是判断题不是必填题
- 用户审完卡片清单提出修改时,**只改卡片不渲染 HTML**,等用户明确确认再进入步骤 5
- 卡片 markdown 文件本身是**独立交付物**(独立的研究归档 + 复用模板),不只是 HTML 的脚手架
- 需要真实截图/照片的位置,用"图槽 ID + 文末图片清单表"表达,**描述只写在清单表**;收图在步骤 4 卡片确认时进行,不拖到渲染后
- 卡片 md 支持用户**离线编辑后回传**;回传版即新的事实来源,渲染时按回传版重新解析(尤其文末图片清单表)
