# Dialog Scripts · 对用户的交流话术

本文件收录本 skill **对用户说的话**(逐字话术)。SKILL.md 主体只保留"在这一步要确认/拒绝/询问什么"的判断逻辑,真正要发给用户的措辞统一放这里。**执行到对应步骤、需要开口跟用户对话时,读本文件取对应话术。**

> 话术可按当下语境微调用词,但**不要改变所询问/告知的实质内容**(尤其是硬门槛、不追问 N/M、只提示一次等约束)。

---

## 1. 启动询问(步骤 1)

一次性合并问三件事(视觉风格 + 业务类型 + 材料盘点),不要拆成三轮。

> 开始之前确认几件事:
>
> **1) 视觉风格**(默认 v4)
> - **v4 现代数据产品感**:多彩当代色 + Geist/JetBrains Mono 字体 + 12 列卡片网格 + SO WHAT/TL;DR 强结论块。适合内部观察、迭代追踪、希望"一目了然"的扫读场景。
> - **v3 沉稳传统感**:深蓝主色 + 米白底 + 屏式长滚动,按公司既有 PPT 模板复刻。适合严肃汇报、对外正式版本。
>
> **2) 业务类型**
> - **ToB**:管理后台、SaaS 工作台、企业流程系统
> - **ToC**:App、小程序、H5、官网、会员与交易流程
> - **ToD**:开发者工具、IDE、API Console、文档站、DevOps
>
> **3) 材料盘点**
>
> **硬门槛(必须有)**:
> - 访谈材料 = **观点聚类文件(JSON 或 Excel)**(由 usability-testing-interview-analysis 产出)
>
> **其他材料(有什么给什么)**:用户信息(被试画像) / 任务书 / 问卷结果 / 观察记录(评测记录表)
> 这 4 类材料**允许缺失或反推**——比如用户信息可能内嵌在问卷题里,任务书可以从观点聚类文件反推任务名等。缺失或反推时,报告中相应位置会显式标注"待补充"。

> **终版/中期不在这里问**——默认按正常报告跑;材料盘点后若检测到被试明显未到齐,才用下面第 2 段提示一次。

---

## 2. 中期快照检测提示(步骤 1.3)

仅在**材料盘点后检测到被试数明显少于计划**时提示**一次**,不展开成长篇,**不主动追问 N/M 具体数值**,用户不理会就按正常报告继续。

> 材料里只看到 N 名被试(计划似乎是 M 名)。要按"中期口径"标注吗?标注后报告会加"初步数据"提示并软化结论措辞。

- 用户确认是中期 → 进入中期口径(口径规则见 `reference/edge-cases.md`「中期快照」)。
- 用户说是终版 / 无明显缺口,或不理会 → 正常跑,不标注。

---

## 3. 拒绝原声逐字稿(步骤 2)

检测到访谈材料是原声逐字稿(或未聚类的原始记录表)时,直接拒绝并简短指引,**不解释设计原因**。

> 这是原始逐字稿,需要先经过观点聚类处理。
>
> 请使用「usability-testing-interview-analysis」处理后,把产出的 JSON 给我,我接着往下做。

> **不要**在拒绝时长篇解释设计原因,**不要**主动给"基于原声的简化分析"降级路径——研究员需要的是"下一步去哪",不是"为什么要这样"。

---

## 4. 粒度确认(步骤 3)

**默认只问这一个**(报告类型);附加产物 / 自定义筛选只在用户提出或场景明显时再展开,不要一次把所有选项抛给用户。

> 你需要什么定位的报告?
> - **完整报告**(默认,推荐用于首次研究归档):从研究设计到附录的全量报告(约 15–25 屏)
> - **重点摘要**:核心结论 + 高风险任务 + 关键问题(约 5–8 屏,适合管理层 / 评审会)
>
> 如果有特殊需求(管理层一页纸、研发清单、自定义筛选),也可以告诉我。

---

## 5. 中间态卡片确认(步骤 4)

提交中间态卡片清单后,用这段征求修改意见。

> 这是报告的中间态卡片清单。请审核后告诉我:
> 1. 哪些屏需要修改观点 / 数据 / 原声 / 图?
> 2. 是否要增加 / 删除某屏?
> 3. 整体顺序是否合理?
>
> 确认无误后,我会渲染成 HTML。如果你只需要卡片清单(不渲染 HTML),也可以告诉我。

**同时征集配图**(图片清单已在卡片 md 末尾列好,逐条对应建议配图的位置)。**话术务必通俗**——别让用户去"理解"`img-p03` 这种 ID,只让他照抄文件名:

> 我在报告里标了几处建议配图的位置,都列在卡片清单最后那张「图片清单」表里。你有对应截图的话,挑省事的方式给我就行:
> - **直接发图**:把截图发我,顺手说一句每张大概是哪个界面 / 对应哪条发现,我来对位。
> - **按名字发**:看表里最左边那列的名字(比如 `img-p03`),把你的截图文件名改成一模一样的(`img-p03.png`)再发我——名字照抄即可,我会自动贴到对应位置。
>
> 没有的图先空着就行,报告里会留一个占位框,以后补了我再放进去。

> **另外告诉用户:卡片清单本身可离线改后回传**:
>
> 这份卡片清单你也可以直接在 .md 文件里改——改文字、调顺序,或者在最后那张表的"对应文件"列里,把你有的截图文件名填到对应行(没有的留空)。改完把文件发回给我,我按你改好的版本接着渲染。

> 用户的修改应是轻量的(改一行字、调一个数字、换一张图引用)。若用户提出**大量**修改,先判断是不是中间态颗粒度有问题,重新和用户对齐颗粒度,而不是埋头逐条改。

---

## 6. 收尾询问(报告产出后)

报告产出后,主动做两件事(不要等用户提):

**a. 对仍缺的配图补问一次**(收图主战场在步骤 4 卡片确认,这里只兜底):报告里若仍有"待补充"的图槽,提醒用户可补——

> 报告里这几处还留着配图占位:[逐项列出占位位置 + 该放什么图]。后面有对应截图了发我(直接发图,或按占位框提示的名字命名),我放进去重出一版。

**b. 主动列"待补充清单"**:把报告中所有标了"待补充 / 未采集"的位置汇总成一份清单告诉用户,方便其补材料后重新生成。

> 这份报告里有以下位置标了"待补充 / 未采集":[逐项列出位置 + 缺什么]。补齐对应材料后告诉我,我重新生成。
