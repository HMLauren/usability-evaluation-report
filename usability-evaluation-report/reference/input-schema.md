# Input Schema · 聚类产物解析规则(JSON / Excel)

usability-testing-interview-analysis 产出的聚类产物是本 skill 最关键的输入之一。本文件定义实际 schema 与解析规则。聚类产物有两种容器格式:**JSON**(本文件大部分章节)与 **Excel**(见末尾「Excel 输入解析规则」),两者解析后归一化为完全相同的内部结构。

> 实际样例文件:`reference/sample-clustering.json`(JSON)、`reference/sample-clustering.xlsx`(Excel)
> 当前版本:基于 2026-05-07 样例

## 顶层结构

JSON 顶层是**数组**,包含 2 个元素:

```json
[
  [...任务列表...],          // 索引 0:观点数据
  { "用户1": "...", "用户2": "..." }  // 索引 1:被试与逐字稿文件路径映射
]
```

| 索引 | 类型 | 内容 |
|---|---|---|
| `data[0]` | array | 任务列表(主数据) |
| `data[1]` | object | 被试 → 逐字稿文件路径映射 |

## 任务对象结构

`data[0]` 数组里每个元素是一个**任务对象**:

```json
{
  "任务名称": "任务一:高校子站的导航入口与首页",
  "任务序号": 1,
  "观点1": { ... },
  "观点2": { ... },
  "观点3": { ... }
}
```

字段说明:

| 字段 | 类型 | 必有 | 说明 |
|---|---|---|---|
| `任务名称` | string | ✓ | 任务全名,通常含"任务一:""任务二:"前缀 |
| `任务序号` | int | ✓ | 1, 2, 3... 与任务名称一致 |
| `观点N` | object | ✓ | N 从 1 开始递增,数量不固定 |

**注意**:`观点N` 的 key 是动态的(`观点1` / `观点2` / ...),解析时按 `观点` 前缀匹配,不能写死。

## 观点对象结构

```json
{
  "观点标签": "导航分类不够直观",
  "观点描述": "现有导航分类...希望使用更直观的人群标签...",
  "原声摘取": {
    "用户1说": "你们下边这个科研工作者,教育工作者,学生开放者,其实还比较直观...",
    "用户2说": "就是你就是你,你这样指向性的说的话..."
  },
  "用户": "用户1:逐字稿-艾先生,用户2:逐字稿-陈先生",
  "用户数": 2,
  "问题类型": "信息设计类",
  "严重程度": "一般"
}
```

字段说明:

| 字段 | 类型 | 必有 | 说明 |
|---|---|---|---|
| `观点标签` | string | ✓ | 简短标签(报告中作为问题/需求的标题) |
| `观点描述` | string | ✓ | 详细描述(报告中作为"问题现象/需求背景") |
| `原声摘取` | object | ✓ | key 动态:`用户N说`,value 是原话字符串 |
| `用户` | string | ✓ | 拼接字符串:`用户N:姓名,用户M:姓名` |
| `用户数` | int | ✓ | 涉及被试数 |
| `问题类型` | string | ✓ | 见下方"问题类型枚举" |
| `严重程度` | string | ✓ | 4 档:致命 / 严重 / 一般 / 提示 |

## 被试映射对象(data[1])

```json
{
  "用户1": "/.../逐字稿-艾先生.docx",
  "用户2": "/.../逐字稿-陈先生.docx"
}
```

key 是被试代号(用户1/用户2/...),value 是该被试逐字稿的原始文件路径。

**用途**:
- 报告中如需追溯原始材料,从此处获取
- 附录"被试材料齐整度登记表"可引用此映射判断哪些被试有逐字稿

## 字段映射:JSON → 报告

### 严重程度映射

上游 4 档字面值在报告中**直接原样显示**,不做术语替换。颜色按表对应:

| 字面值(报告中显示) | 颜色 token |
|---|---|
| `致命` | `--sev-fatal` |
| `严重` | `--sev-serious` |
| `一般` | `--sev-normal` |
| `提示` | `--sev-hint` |

⚠️ 不要把字面值替换为"危急/中等/轻微"等其他术语 —— v3/v4 两版均保持上游 schema 原值,降低对齐成本与歧义。

旧别名 `--danger / --warning / --info` 在 design-tokens.md 中仍保留作兼容,但新代码请直接用 `--sev-*` 语义 token。

### 问题类型枚举

已知问题类型(正式枚举,共 7 类):

- `视觉设计类`
- `信息设计类`
- `交互操作类`
- `系统性能类`
- `客户服务类`
- `业务流程类`
- `产品功能类`

直接用作"归属"字段,不做转换。

如出现以上 7 类之外的未知类型,在报告附录的"数据口径说明"中标注:"出现新问题类型 XXX,未在已知枚举中"。

### 被试 ID 标准化

JSON 中被试代号是 `用户1 / 用户2 / ...`,**报告中统一显示为 `P01 / P02 / ...`**。

转换规则:
- `用户1` → `P01`
- `用户2` → `P02`
- ...
- `用户N` → `P` + N 的两位补零字符串

实现:

```python
import re
def normalize_user_id(s):
    """'用户1' → 'P01'"""
    m = re.match(r'用户(\d+)', s)
    if m:
        return 'P' + m.group(1).zfill(2)
    return s
```

### 用户字段解析

`"用户1:逐字稿-艾先生,用户2:逐字稿-陈先生"` 这种字符串需要拆解。

解析规则:

```python
def parse_users(s):
    """
    输入: '用户1:逐字稿-艾先生,用户2:逐字稿-陈先生'
    输出: [
      {'id_raw': '用户1', 'id': 'P01', 'name': '艾先生', 'source': '逐字稿'},
      {'id_raw': '用户2', 'id': 'P02', 'name': '陈先生', 'source': '逐字稿'}
    ]
    """
    result = []
    for item in s.split(','):
        item = item.strip()
        # 形如: "用户1:逐字稿-艾先生"
        match = re.match(r'(用户\d+)[::](.+?)-(.+)', item)
        if match:
            id_raw, source, name = match.groups()
            result.append({
                'id_raw': id_raw,
                'id': normalize_user_id(id_raw),
                'source': source,
                'name': name
            })
    return result
```

### 原声摘取解析

key 是 `XXX说` 的格式,需要按模式匹配:

```python
def parse_quotes(quotes_dict):
    """
    输入: {'用户1说': '...', '用户2说': '...'}
    输出: [
      {'user_id': 'P01', 'quote': '...'},
      {'user_id': 'P02', 'quote': '...'}
    ]
    """
    result = []
    for key, value in quotes_dict.items():
        # 提取"用户N"部分
        match = re.match(r'(用户\d+)说', key)
        if match:
            result.append({
                'user_id': normalize_user_id(match.group(1)),
                'quote': value
            })
    return result
```

### 观点 ID 生成

JSON 没有给观点 ID,模型自己生成。规则:

**格式**:`T{任务序号}-OP{观点在该任务内的序号}`

示例:
- 任务1 的 观点1 → `T1-OP1`
- 任务6 的 观点8 → `T6-OP8`

如果报告需要全局唯一的 P-XXX 编号(用于"问题深挖"和附录),用全局递增序号:

**全局编号规则**:按"严重程度优先级 + 任务序号 + 观点序号"排序,然后从 P-001 开始递增。

排序优先级:致命(0) > 严重(1) > 一般(2) > 提示(3)

```python
SEVERITY_ORDER = {'致命': 0, '严重': 1, '一般': 2, '提示': 3}

def assign_global_ids(opinions):
    """opinions: [{'task_id': 1, 'op_idx': 1, 'severity': '致命', ...}, ...]"""
    sorted_ops = sorted(opinions, key=lambda x: (
        SEVERITY_ORDER.get(x['severity'], 99),
        x['task_id'],
        x['op_idx']
    ))
    for i, op in enumerate(sorted_ops, 1):
        op['global_id'] = 'P-' + str(i).zfill(3)
        op['local_id'] = f"T{op['task_id']}-OP{op['op_idx']}"
    return sorted_ops
```

报告中:
- 顶部"问题切换 tab"用 `P-001 / P-002 / ...`(按严重等级降序排)
- 元信息条里附 `T1-OP1` 作为"任务来源溯源 ID"

## 边界情况

### 1. JSON 顶层只有 1 个元素(没有被试映射)

跳过 `data[1]` 的解析,但报告中"附录·被试材料齐整度登记"无法显示文件路径,只显示被试 ID + 姓名。

### 2. 某个观点的"原声摘取"为空 dict 或缺失

跳过该观点的"用户原声"区块,仍保留观点其他信息。在报告中标注"(无可引用原声)"。

### 3. "用户"字段格式异常

如果正则匹配失败,**不阻断流程**,把字符串原样存为 `name`,id 标为 `?`。在报告附录"数据口径说明"中标注异常被试。

### 4. 严重程度出现新值(如"致命+"或英文)

按未知值处理:颜色用 `--text-2`(neutral),报告附录"数据口径说明"中标注。

### 5. 同一被试在多个观点中出现

正常,这是聚类的预期结果。报告中:
- 该被试的不同原话分别被引用
- 在"问题深挖"页中,如果一个问题影响 5+ 被试,优先选**多被试**的原话(同一被试最多 1 条)

### 6. 任务序号不连续(跳号)

正常处理,不假设序号连续。

以下问题待确认,目前按合理默认值处理:

# Excel 输入解析规则

聚类产物的第二种容器格式。结构上是**一行一个观点的平铺表**,解析后产出与上文 JSON 完全相同的内部结构(任务列表 + 观点对象 + 被试映射),供步骤 3/4/5 无差别消费。

> 实际样例文件:`reference/sample-clustering.xlsx`

## 表头与列映射

第一行是表头,标准 8 列(列名按字面匹配,允许前后空白):

| Excel 列 | 对应 JSON 字段 | 说明 |
|---|---|---|
| `任务名称` | `任务名称` | 合并单元格式,仅每组首行有值(见「分组重建」) |
| `观点标签` | `观点标签` | 观点标题 |
| `观点描述` | `观点描述` | 观点详述 |
| `原声摘取` | `原声摘取` | 单格多条,见「原声单元格解析」 |
| `用户` | `用户` | 见「用户列解析」,格式与 JSON 略有差异 |
| `用户数` | `用户数` | 整数,空/非数字时回退为解析出的用户数 |
| `问题类型` | `问题类型` | 6 类枚举,未知值按上文规则标注 |
| `严重程度` | `严重程度` | 致命 / 严重 / 一般 / 提示 |

**列名容错**:若列名与上表不完全一致(如 `观点` / `标签` / `问题描述` / `原话` / `严重度`),做一次自动近似映射;映射不确定时,向用户确认列对应关系,不擅自猜。

**没有的列**:Excel 通常不含 `任务序号`、`观点N` key、独立的被试映射 sheet —— 这些都由解析过程生成或重建(见下)。

## 分组重建(关键)

`任务名称` 是合并单元格写法:同一任务的多个观点中,只有**第一行**有任务名,后续行该格**为空**。

```python
# 向下填充:空任务名继承上一行
last_task = None
for row in rows:
    name = (row.get('任务名称') or '').strip()
    if name:
        last_task = name
    row['_task'] = last_task
```

- 连续同名行归为一个任务;按**首次出现顺序**生成 `任务序号`(1,2,3…)
- 任务内观点按**行序**生成观点序号(`观点1` / `观点2` …),对应 JSON 的 `观点N` key
- 注意:这条规则假设「空任务名 = 同上一任务」。若上游 Excel 用了别的留空含义,需先与用户对齐

## 原声单元格解析

`原声摘取` 一个单元格内塞多条原话,每条以 `用户N说:` 起头(真实换行分隔)。**同一用户可有多条**,第 2 条起带 `_N` 后缀(`用户1说` / `用户1说_2` / `用户1说_3`):

```
用户2说: <原话…>
用户3说: <原话…>
用户3说_2: <同一人的第二段…>
```

```python
import re
QUOTE_DELIM = r'用户\d+说(?:_\d+)?[:：]'   # 允许 _2/_3 等后缀,半/全角冒号
def parse_quote_cell(cell):
    """单格多条 → {'P02': '...', 'P03': '...'}(同一人多条自动合并)"""
    if not cell or not str(cell).strip():
        return {}
    parts = re.split(f'({QUOTE_DELIM})', str(cell))
    result, cur = {}, None
    for seg in parts:
        if re.match(QUOTE_DELIM, seg):
            cur = 'P' + re.match(r'用户(\d+)', seg).group(1).zfill(2)
        elif cur:
            merged = (result.get(cur, '') + ' ' + seg).strip()
            result[cur] = merged
    # 丢弃占位/空值(如 "无"、空串——聚类时偶有占位)
    return {k: v for k, v in result.items() if v and v.strip() not in ('无',)}
```

- 半角 `:` 与全角 `：` 都要兼容;`用户N说_2` 这类后缀**必须**识别,否则会把同人第二段当成上一条的正文
- 同一用户多条 → **合并**为一条(报告里一个被试在一个问题下不重复列多段)
- 某条原话值为 `无` / 空 → 视为该用户无引用,丢弃(聚类阶段偶尔留占位)
- 单条原话有时**不带 `用户N说:` 前缀**(整格就是一句话)→ 回退:若该行 `用户` 列只有 1 人,把整格归给该用户;多人则标 `(原声归属待确认)`,不阻断
- 整格空格 / 缺失 → 该观点标「(无可引用原声)」,同 JSON 规则

## 用户列解析(与 JSON 有差异)

JSON 是 `用户N:逐字稿-姓名`(含来源段,`-` 是分隔符);**Excel 的标识段不拆**——里面的 `-` 往往是 label 自带的(平台前缀如 `抖音-唐湘润`、`B站-王帅强`,或内部编号如 `IDEB站1-1`),按 `-` 强拆会把姓名拆错:

```
用户2：艾先生,用户3：陈先生              # 纯姓名
用户5：抖音-唐湘润,用户7：B站-王帅强      # 平台-姓名:整段当 label,不拆
用户3：IDEB站1-1,用户14：黑猫IDE         # 内部编号/代号:整段当 label
```

因此**按格式分流**:JSON 模式拆 `源-名`,Excel 模式整段作 label。

```python
def parse_users(s, json_mode=False):
    """json_mode=True 拆 '逐字稿-姓名';Excel(默认)整段作 label,不按 - 拆"""
    result = []
    for item in re.split(r'[,，]', str(s or '')):
        item = item.strip()
        m = re.match(r'(用户\d+)[：:](.+)', item)
        if not m:
            continue
        id_raw, rest = m.group(1), m.group(2).strip()
        if json_mode and '-' in rest:     # 仅 JSON:源-名
            source, name = rest.split('-', 1)
        else:                             # Excel:整段就是标识,不拆连字符
            source, name = None, rest
        result.append({'id_raw': id_raw, 'id': normalize_user_id(id_raw),
                       'source': source, 'name': name})
    return result
```

> **可选增强(不默认做)**:平台前缀(抖音/B站/小红书/美团…)其实是有价值的招募来源元数据,可进 ToD 用户画像。但因 label 格式不统一(`IDEB站1-1`、`黑猫IDE` 不符合「平台-姓名」),**不做自动拆分**;如需结构化平台字段,建议上游在 Excel 单列里单独给,而非从 `用户` 列硬猜。

## 被试映射重建(无独立 sheet 时)

Excel 通常没有 JSON `data[1]` 那样的被试→逐字稿路径映射。处理:

- **从全表 `用户` 列聚合**,重建「被试 ID → 姓名」表(如 `用户1→姚工 / 用户2→艾先生 / …`)
- 逐字稿**文件路径标缺失**,附录「数据口径说明」注明「被试映射由 Excel 用户列重建,无原始逐字稿路径」
- 若同一 `用户N` 在不同行出现不同姓名 → 取首次出现值,在附录标注冲突,不阻断

## 交叉校验(不阻断,只标注)

| 校验 | 不一致时 |
|---|---|
| `用户数` vs 实际解析出的用户人数 | 以解析人数为准,附录标注差异 |
| `原声摘取` 里的 `用户N` vs `用户` 列的 `用户N` | 取并集;原声中多出的人补进用户表,附录标注 |
| `问题类型` 是否在 6 类枚举内 | 未知值按上文「问题类型枚举」规则标注 |
| `严重程度` 是否在 4 档内 | 未知值按边界情况 §4 处理 |

## 归一化产出

上述步骤产出的内部结构与 JSON 解析后**逐字段一致**:任务列表(含 `任务序号` / `观点N`)+ 观点对象(七字段)+ 被试映射(重建版)。此后的 ID 生成、严重程度映射、报告渲染等全部复用 JSON 路径,**不区分原始格式**。
