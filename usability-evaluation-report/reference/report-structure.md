# Report Structure · 报告章节结构与写作规格

本文件定义报告的章节屏级结构、SO WHAT 写作规范、数据/原声引用格式、衍生产物筛选规格。**渲染报告前必读。** 所有内容须遵守 SKILL.md「报告内容必须有据」的防幻觉底线。

## 报告章节模板

按公司现有 PPT 模板的章节结构落地,共 5 个主章节:

### 第 0 章 · 封面与版本

- 屏 0.0:封面(产品名、责任人、部门、版本号、更新时间、封面图)
  - **H1 主标题 = 报告主题(产品名 + 评估类型,如「XX 改版可用性评估报告」),禁止用结论句当主标题。** 卡片清单屏 0.0 的「核心观点/结论」是结论,降级为副标题(主标题下一行,关键词高亮)或并入 TL;DR,不占主标题位。
- 屏 0.1:版本修订记录(可选,有迭代时才出)
- 屏 0.2:目录(可选,自动生成,默认嵌入顶部导航而非独立屏)

### 第 1 章 · 研究设计

- 屏 1.0:章节扉页
- 屏 1.1:项目背景(研究目的、方法、采样信息)
- 屏 1.2:评测场景与任务(任务清单 + 来源依据)
- 屏 1.3:招募样本与样本充分性(用户画像 + 尼尔森公式)

### 第 2 章 · 整体测试结果

- 屏 2.0:章节扉页
- 屏 2.1:整体结论(结论横幅/SO WHAT 卡 + 维度均分雷达 + 问题/需求接纳率)
- 屏 2.2:任务维度分析(完成率 / 出错次数 / 满意度,按任务呈现)
- 屏 2.3:问题维度分析(严重等级分布 + 问题类型分布,**章末 SO WHAT**)
- 屏 2.4:需求分析(需求清单 + 优先级)

### 第 3 章 · 易用性问题深挖

- 屏 3.0:章节扉页(声明"仅展开致命/严重问题")
- 屏 3.x:每个致命/严重问题独立一屏,结构:
  - 问题切换 tab(顶部)
  - 标题 + 严重等级标签
  - 元信息条(任务、影响人数、归属、主责、状态)
  - 左右双栏:问题现象/根因 | 用户原声
  - (可选)问题现象处放图槽展示界面截图——正文只放图槽 ID(如 `img-p03`),描述写进卡片文末「图片清单」表;缺图渲占位。详见 `reference/card-templates.md` 与 `reference/screen-layouts.md`「图片渲染与降级」
  - 优化建议(P0/P1/P2 短中长期)
  - 验收标准
  - **SO WHAT(仅高频/严重问题)**

### 第 4 章 · 体验需求深挖

- 屏 4.0:章节扉页
- 屏 4.x:每个高价值需求独立一屏,结构:
  - 需求切换 tab(顶部)
  - 标题 + 价值标签
  - 元信息条(任务、提出人数、类别)
  - 需求背景 / 用户诉求 / 机会点
- 章末 SO WHAT

### 第 5 章 · 附录

- 屏 5.0:章节扉页
- 屏 5.1:问题清单总表(全量,可筛选)
- 屏 5.2:需求清单总表(全量)
- 屏 5.3:被试材料齐整度登记表
- 屏 5.4:严重度分级说明
- 屏 5.5:数据口径说明

## SO WHAT 段落规范

SO WHAT 是 v4 卡片网格的核心特色,也在 v3 中以"结论横幅"组件承载——它把数据观察转化为业务行动方向,是研究报告从"陈述"走向"决策"的关键。

### 触发位置

**必须**写 SO WHAT 的位置:

1. **每章末尾**(除第 0 章封面、第 5 章附录) —— 由该章末屏承担
2. **整体结论卡(屏 2.1)** —— 全报告的核心 SO WHAT
3. **高频/严重问题卡** —— 严重等级 = 致命/严重 **且** 影响人数 ≥ 总被试数的 50%

**不写** SO WHAT 的位置:封面、章节扉页、附录、低频问题/需求(汇总进表格)、画像卡、背景卡。

### 写作公式

```
[数据观察] → [业务含义] → [行动方向]
```

控制在 **20-40 字**,一句话。

### 示例

| 场景 | SO WHAT |
|---|---|
| 整体结论卡(屏 2.1) | "整体均分 6.8 → 需求接纳率 73% 是积极信号,但信息架构维度 5.2 是 P0 修复方向" |
| 问题维度卡(屏 2.3 章末) | "信息设计类问题占 47% → 改版后的导航信息架构是优先级 P0 的修复方向" |
| 高频问题卡 | "10/12 被试找不到入口 → 一级导航的可见性是迭代 1.0 的必修项" |
| 需求章末 | "62% 被试提到批量操作 → 与现有单条流程并存,而非替代" |

### 视觉承载

- **v4**:专用 SO WHAT 卡组件(淡琥珀底 + 关键词高亮),见 sample-mockup-v4.html
- **v3**:用"结论横幅"组件承载(深蓝底 + 白字),内容文本相同,只是视觉皮肤不同

## 数据与原声引用规则

### 数据引用

- 所有定量数据必须**有来源**,在图表下方或表格脚注标注:`数据来源:XX问卷 · n=12 · 10点量表 · 取均值`
- 数据格式统一:百分比保留整数(如 38%),均分保留 1 位小数(如 7.5)
- 跨章节数据必须一致——结果总览说"23 个问题",附录就必须列出 23 行

### 原声引用

- **逐字引用**:原声必须**一字不改**地来自观点聚类文件的 `原声摘取` 字段,不得改写、补全、润色或合并语义;**宁可少引,绝不编引**(详见「报告内容必须有据」)
- 每条原声必须标注:**被试 ID + 时间戳**(如 `P02 · 12:34`)。时间戳上游没给就只标被试 ID,不要编一个
- 一个问题/需求页中,**最多引用 3 条原声**(过多稀释重点,过少缺乏代表性);源里不足 3 条就少放
- 优先选择:陈述清晰、有情感色彩、能代表多人观点的原话
- 避免选择:语义模糊、过于个人化、需要大量上下文才能理解的片段

### 数据与原声的关联

- 当某个问题影响 5+ 被试时,原声引用应**覆盖至少 2 个不同被试**(不能 3 条都来自同一人)
- 当结论是"高频问题"时,数据上必须有支撑(如"6/12 被试遇到此问题")

### 被试维度的材料覆盖度登记

不同被试在访谈材料中的覆盖度可能不同——某些被试观点丰富,某些被试只有少量原话被聚类提取。这是聚类阶段的自然结果。

报告附录会生成"被试材料覆盖度登记表",示例:

| 被试 ID | 在观点聚类文件中出现的观点数 | 观察记录 | 备注 |
|---|---|---|---|
| P01 | 12 个观点 | ✓ | 高活跃 |
| P02 | 8 个观点 | ✓ | |
| P03 | 3 个观点 | ✗ | 访谈时间较短 |

这张表帮助报告读者理解每个被试对报告内容的贡献度,也是「观点高度集中在某一被试」边界场景判断的依据。

## 衍生产物筛选规则(管理层一页纸 / 研发清单 / 与上期对比)

附加产物**不重新生成数据**,而是**从同一份中间态卡片清单中筛取**(确保数据源一致)。

**管理层一页纸**(`-summary-exec.html`):

从中间态卡片中抽取:
- 屏 0.0 封面元信息(产品名、版本)
- 屏 2.1 整体结论(核心一句话 + 关键大数字:问题数 / 需求数 / 接纳率 / 满意度均分)
- 屏 3.x 中**严重等级 = 致命/严重 且影响人数前 3 的问题**(只取标题 + 一句话描述,不展开)
- 屏 2.3 问题类型分布(简化为前 3 类百分比)

不包含:用户画像、维度均分图、用户原声、需求深挖、附录。

设计取向:信息密度低、视觉醒目、3 分钟可读完、单屏不滚动。

**研发清单**(`-issues-eng.html`):

从中间态卡片中抽取:
- **所有问题卡(屏 3.x)** → 字段:问题 ID / 严重等级 / 描述 / 涉及任务 / 影响人数 / 问题类型 / 优化建议(P0/P1/P2 三栏) / 验收标准
- **所有需求卡(屏 4.x)** → 字段:需求 ID / 价值标签 / 描述 / 涉及任务 / 提出人数 / 类别

主体是两张大表格(问题表 + 需求表)+ 顶部筛选栏。默认按"严重等级 → 影响人数"双重排序(问题表)、"价值标签 → 提出人数"排序(需求表)。

不包含:研究背景、用户画像、用户原声、设计层面的根因分析。

设计取向:可执行、可追溯、便于导入研发工单系统。

**与上期对比**:

在主报告基础上,在屏 2.1 整体结论、屏 2.3 问题维度分析等位置插入对比数据条(本期 vs 上期),用箭头/颜色标注变化方向。不单独成文件,合并进主报告。
